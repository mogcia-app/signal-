# タブ切り替え vs サイドバー独立：比較

## 選択肢1: タブ切り替え（同じページ内）

### 構成
```
/instagram/report
├─ タブ1: パフォーマンス評価
└─ タブ2: KPIコンソール
```

### メリット
- ✅ 1つのページで完結
- ✅ 状態管理が簡単（タブの状態だけ）
- ✅ URLが1つ（シンプル）
- ✅ データを共有しやすい

### デメリット
- ❌ ページが複雑になる可能性
- ❌ 既存の `/instagram/monthly-report` と同じ構造になる
- ❌ タブ間でデータを保持する必要がある

---

## 選択肢2: サイドバーに独立（別ページ）

### 構成
```
/instagram/report          # パフォーマンス評価ページ
/instagram/report/kpi      # KPIコンソールページ
```

### メリット
- ✅ **各ページがシンプル**（責務が明確）
- ✅ **独立して開発できる**（並行作業可能）
- ✅ **URLが明確**（ブックマーク、共有しやすい）
- ✅ **既存の複雑なタブ切り替えを避けられる**
- ✅ **将来的に拡張しやすい**（新しいページを追加しやすい）
- ✅ **パフォーマンスが良い**（必要なデータのみ読み込む）

### デメリット
- ❌ ページ遷移が必要
- ❌ データの共有が少し複雑（必要に応じて）

---

## 推奨：サイドバーに独立（選択肢2）

### 理由

1. **シンプルさを優先**
   - 既存の `/instagram/monthly-report` はタブ切り替えで複雑になった
   - 新規ページはシンプルに保つべき

2. **責務の明確化**
   - パフォーマンス評価 = 評価スコアとPDCA実行度
   - KPIコンソール = 詳細なKPI分析
   - それぞれ独立したページの方が明確

3. **開発の柔軟性**
   - 各ページを独立して開発できる
   - テストが容易
   - デプロイも独立できる

4. **将来的な拡張**
   - 新しいページを追加しやすい（例: `/instagram/report/trends`）
   - サイドバーにメニューを追加するだけ

### 実装イメージ

#### サイドバー構造
```
📊 レポート
  ├─ 📈 パフォーマンス評価 (/instagram/report)
  └─ 📉 KPIコンソール (/instagram/report/kpi)
```

#### ページ構成
```
/instagram/report/
├── page.tsx                    # パフォーマンス評価ページ
├── components/
│   ├── ReportHeader.tsx
│   ├── PerformanceScore.tsx
│   └── PDCAExecutionRate.tsx
└── kpi/
    ├── page.tsx                # KPIコンソールページ
    └── components/
        ├── KPICards.tsx
        ├── KPIDrilldown.tsx
        └── DetailedStats.tsx
```

## 実装ステップ

### フェーズ1: パフォーマンス評価ページ
- `/instagram/report` を完成
- パフォーマンス評価スコア（A〜F）
- PDCA実行度（％）

### フェーズ2: KPIコンソールページ
- `/instagram/report/kpi` を作成
- 基本的なKPI表示
- 前月比表示

### フェーズ3: サイドバー更新
- サイドバーに「レポート」セクションを追加
- 2つのページへのリンクを追加

---

## 結論

**サイドバーに独立させることを推奨します。**

理由：
1. シンプルさを優先
2. 責務が明確
3. 開発が柔軟
4. 将来的に拡張しやすい
