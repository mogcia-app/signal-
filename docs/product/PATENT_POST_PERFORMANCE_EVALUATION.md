# ソーシャルメディア投稿のパフォーマンス評価システムの特許申請資料

## （1）従来技術

### 1.1 Instagram公式Analytics（Instagram Insights）

**概要**：
Instagram公式が提供する分析機能。投稿ごとのいいね数、リーチ数、エンゲージメント率などの基本メトリクスを表示する。

**URL**: https://business.instagram.com/insights

**機能**：
- 投稿ごとのメトリクス表示（いいね数、コメント数、リーチ数、保存数など）
- 期間別の集計データ表示
- グラフによる可視化

**問題点**：
- 各メトリクスの絶対値のみを表示し、相対的な評価は行わない
- 複数の指標を統合した総合評価スコアは算出しない
- 投稿を「成功/失敗」などのカテゴリに自動分類する機能がない
- 主観的な満足度（ユーザーフィードバック）と客観的な指標を組み合わせた評価は行わない

---

### 1.2 SNS分析ツール（Sprout Social、Hootsuite等）

**概要**：
複数のソーシャルメディアアカウントを一元管理するツール。投稿スケジューリング、分析機能、エンゲージメント管理などを提供。

**代表的なサービス**：
- Sprout Social: https://sproutsocial.com/
- Hootsuite: https://www.hootsuite.com/

**機能**：
- 複数プラットフォーム対応
- エンゲージメント分析
- レポート生成
- データの可視化

**問題点**：
- 各メトリクスを個別に表示するのみで、複数指標を統合した評価スコアは算出しない
- 投稿のパフォーマンスを「成功/失敗」などに自動分類する機能がない
- ベースライン（過去平均値）との相対評価は行わない
- 主観的満足度と客観的指標を組み合わせた評価システムがない

---

### 1.3 一般的なアナリティクスツール

**概要**：
Webサイトやアプリのアクセス解析、SNS分析に特化したツール。データの可視化やレポート生成を主な機能とする。

**代表的なサービス**：
- Google Analytics（Web分析）
- Twitter Analytics（Twitter公式）

**機能**：
- データ収集と集計
- グラフ・チャートによる可視化
- レポート生成
- 期間比較機能

**問題点**：
- メトリクスの絶対値のみを表示し、相対評価は行わない
- 複数指標を統合したスコア計算は行わない
- 投稿を自動分類する機能がない
- 主観的満足度と客観的指標の統合評価は行わない

---

### 1.4 従来技術の共通点

1. **絶対値のみの表示**：各メトリクスの絶対値（いいね数、リーチ数など）を表示するが、相対的な評価は行わない
2. **統合評価スコアの欠如**：複数の指標（リーチ、エンゲージメント率、フォロワー増加など）を統合した総合評価スコアは算出しない
3. **自動分類機能の欠如**：投稿を「成功/失敗」「良い/悪い」などのカテゴリに自動分類する機能がない
4. **主観と客観の分離**：主観的な満足度（ユーザーフィードバック）と客観的な指標（KPI）を組み合わせた評価システムがない
5. **ベースライン比較の限界**：過去データとの比較はあるが、ベースライン（平均値）との相対評価による判定ロジックは実装されていない

## （2）問題点

従来技術には、以下のような問題点が存在する。

### 2.1 複数の指標を統合評価できない

**問題点の詳細**：
- 従来技術は、リーチ数、エンゲージメント率、フォロワー増加など、各指標を個別に表示することはできるが、これらの指標を統合した総合評価スコアは算出しない
- ユーザーは複数の指標を見比べて、投稿のパフォーマンスを総合的に判断する必要がある
- どの指標が重要か、どの指標を優先すべきかはユーザーの経験や知識に依存する

**具体例**：
- リーチ数は高いがエンゲージメント率が低い投稿と、リーチ数は低いがエンゲージメント率が高い投稿のどちらが「良い」投稿か判断が困難
- フォロワーが増加したが、エンゲージメント率が低下した場合の総合評価が不明確

---

### 2.2 ベースラインとの相対評価ができない

**問題点の詳細**：
- 従来技術は、各メトリクスの絶対値のみを表示し、そのアカウントの過去平均値（ベースライン）との相対評価は行わない
- アカウント規模やフォロワー数によって、同じいいね数でも意味が異なるため、絶対値だけでは適切な評価ができない
- 過去の投稿と比較して「良い/悪い」を判断するには、ユーザーが手動で過去データを参照する必要がある

**具体例**：
- 100いいねを獲得した投稿が「良い」か「悪い」かは、そのアカウントの過去平均が50いいねなのか500いいねなのかによって異なる
- 小さなアカウントと大きなアカウントで、同じいいね数を比較しても意味がない

---

### 2.3 投稿を自動分類できない

**問題点の詳細**：
- 従来技術は、投稿のメトリクスを表示することはできるが、投稿を「成功パターン」「改善が必要」「参考レベル」などのカテゴリに自動分類する機能がない
- ユーザーは、表示されたデータを見て、自分で投稿の良し悪しを判断する必要がある
- 複数の投稿を比較して、どの投稿が成功パターンかを識別する作業は手動で行う必要がある

**具体例**：
- 月に20件の投稿がある場合、どの投稿が「成功パターン」で、どの投稿が「改善が必要」かを、すべてのデータを見比べて判断する必要がある
- 成功パターンの投稿を学習して、次の投稿に活かすには、手動でパターンを抽出する必要がある

---

### 2.4 主観的満足度と客観的指標を統合評価できない

**問題点の詳細**：
- 従来技術は、客観的な指標（いいね数、リーチ数など）は表示するが、主観的な満足度（ユーザーフィードバック、コメントの感情など）と組み合わせた評価は行わない
- 数値は高いがユーザーからの評価が低い投稿と、数値は低いがユーザーからの評価が高い投稿を区別できない
- 主観と客観のバランスを考慮した評価システムがない

**具体例**：
- エンゲージメント率は高いが、ユーザーからのフィードバックが「不満」である投稿を、「良い」投稿として評価してしまう可能性がある
- 逆に、エンゲージメント率は低いが、ユーザーからのフィードバックが「満足」である投稿を、「悪い」投稿として評価してしまう可能性がある

---

### 2.5 再現可能性や改善可能性を評価できない

**問題点の詳細**：
- 従来技術は、投稿のパフォーマンスを数値で表示することはできるが、その投稿が「次も再現できるパターンがあるか（再現可能性）」「改善すれば変えられる要因が特定できたか（改善可能性）」を評価する機能がない
- 数値が高い投稿でも、それが再現可能なパターンなのか、単に偶然なのかを区別できない
- 数値が低い投稿でも、改善可能な要因が特定できる投稿と、改善困難な投稿を区別できない

**具体例**：
- エンゲージメント率が高い投稿が、「次も再現できる明確なパターンがある投稿」なのか、「偶然うまくいった投稿」なのか判断できない
- エンゲージメント率が低い投稿が、「ハッシュタグを追加すれば改善できる投稿」なのか、「根本的に改善困難な投稿」なのか判断できない

---

### 2.6 閾値による判定ロジックがない

**問題点の詳細**：
- 従来技術は、メトリクスを表示することはできるが、「ベースラインの1.3倍以上なら高パフォーマンス」「ベースラインの0.7倍以下なら低パフォーマンス」といった、閾値による自動判定ロジックは実装されていない
- ユーザーは、表示された数値を見て、自分で「良い/悪い」を判断する必要がある
- 判断基準が主観的になりがちで、一貫性のない評価になる可能性がある

**具体例**：
- エンゲージメント率が5%の投稿が「良い」か「悪い」かは、そのアカウントの平均が3%なのか7%なのかによって異なるが、従来技術ではこの判断を自動化できない

---

## （3）発明内容

本発明は、ソーシャルメディア投稿の複数の指標を統合し、ベースラインとの相対評価によりKPIスコアを算出し、主観的満足度と客観的指標を組み合わせて投稿を4段階（gold/gray/red/neutral）に自動分類するシステムである。

### 3.1 KPIスコア計算アルゴリズム

**発明の構成要素**：

1. **ベースラインメトリクス計算部**：
   - 過去の投稿データから、平均リーチ数、平均エンゲージメント率、平均フォロワー増加数を計算
   - 各投稿のベースライン（基準値）を算出

2. **相対評価計算部**：
   - 各投稿のリーチ数、エンゲージメント率、フォロワー増加数を、ベースラインで割って相対評価を算出
   - 各指標の相対評価値を配列に格納

3. **KPIスコア算出部**：
   - 各指標の相対評価値の平均を計算
   - 0-3の範囲に正規化（Math.max(0, Math.min(3, 平均値))）
   - KPIスコアとして返却

**技術的特徴**：
- 複数の指標（リーチ、エンゲージメント率、フォロワー増加）を統合した単一スコアを算出
- ベースライン（過去平均値）との相対評価により、アカウント規模に依存しない評価を実現
- 0-3の範囲に正規化することで、スコアの解釈を容易にする

**実装例**：
```typescript
// ベースラインとの相対評価
const components: number[] = [];
if (baseline.avgReach > 0) {
  components.push(record.reach / baseline.avgReach);
}
if (baseline.avgEngagement > 0) {
  components.push(engagementRate / baseline.avgEngagement);
}
if (baseline.avgFollowerIncrease > 0) {
  components.push(
    (record.followerIncrease + 0.01) / (baseline.avgFollowerIncrease + 0.01)
  );
}

// KPIスコアを0-3の範囲に正規化
const kpiScore = components.length > 0
  ? Math.max(0, Math.min(3, components.reduce((sum, value) => sum + value, 0) / components.length))
  : 0;
```

---

### 3.2 highKpi/lowKpi判定ロジック

**発明の構成要素**：

1. **高パフォーマンス判定部（highKpi）**：
   - KPIスコアが1.1以上
   - または、エンゲージメント率がベースラインの1.3倍以上
   - または、リーチ数がベースラインの1.3倍以上
   - または、フォロワー増加が0より大きい
   - 上記のいずれかの条件を満たす場合、highKpi = true

2. **低パフォーマンス判定部（lowKpi）**：
   - KPIスコアが0.85以下
   - または、エンゲージメント率がベースラインの0.7倍以下
   - または、リーチ数がベースラインの0.7倍以下
   - 上記のいずれかの条件を満たす場合、lowKpi = true

**技術的特徴**：
- 1.3倍/0.7倍の閾値による明確な判定基準
- 複数条件のOR判定による柔軟な評価
- ベースラインとの相対評価により、アカウント規模に依存しない判定を実現

**実装例**：
```typescript
const highKpi =
  kpiScore >= 1.1 ||
  (baseline.avgEngagement > 0 && engagementRate >= baseline.avgEngagement * 1.3) ||
  (baseline.avgReach > 0 && record.reach >= baseline.avgReach * 1.3) ||
  record.followerIncrease > 0;

const lowKpi =
  (kpiScore > 0 && kpiScore <= 0.85) ||
  (baseline.avgEngagement > 0 && engagementRate <= baseline.avgEngagement * 0.7) ||
  (baseline.avgReach > 0 && record.reach <= baseline.avgReach * 0.7);
```

### 3.3 感情スコア計算アルゴリズム

**発明の構成要素**：

1. **フィードバック集約部**：
   - ユーザーからのフィードバック（満足/不満/中立）を集約
   - 各フィードバックに重み付け（positiveWeight, negativeWeight, neutralWeight）を付与

2. **感情スコア算出部**：
   - 感情スコア = positiveWeight / (positiveWeight + negativeWeight + neutralWeight)
   - 0-1の範囲のスコアとして算出
   - 感情ラベル（positive/negative/neutral）を付与

**技術的特徴**：
- フィードバックの重み付けによる感情スコア計算
- 0-1の範囲に正規化されたスコア
- 主観的な満足度を数値化

**実装例**：
```typescript
const totalWeight = positiveWeight + negativeWeight + neutralWeight;
const sentimentScore = totalWeight > 0 ? positiveWeight / totalWeight : 0.5;

let label: "positive" | "negative" | "neutral" = "neutral";
if (sentimentScore >= 0.6) {
  label = "positive";
} else if (sentimentScore <= 0.4) {
  label = "negative";
}
```

---

### 3.4 再現可能性スコア・改善可能性スコア計算アルゴリズム

**発明の構成要素**：

1. **再現可能性スコア計算部（calculateReproducibilityScore）**：
   - フィードバックで「再現したい」という意見があるかチェック（positiveRatio >= 0.7で0.3点）
   - コメントに「また見たい」「参考になる」などのキーワードがあるかチェック（0.2点）
   - 類似投稿（同じカテゴリ）との一貫性をチェック（0.2点）
   - ハッシュタグパターンが構造化されているかチェック（0.15点）
   - タイトルやコンテンツが明確なテーマを持っているかチェック（0.15点）
   - 0-1の範囲に正規化

2. **改善可能性スコア計算部（calculateImprovabilityScore）**：
   - フィードバックで改善点が指摘されているかチェック（negativeRatio >= 0.3で0.3点）
   - コメントに改善を促すキーワードがあるかチェック（0.2点）
   - 改善可能な要因（ハッシュタグが少ない、タイトルが不明確など）が特定できるかチェック（0.2-0.15点）
   - 0-1の範囲に正規化

**技術的特徴**：
- 数値だけでなく、「次も再現できるか」「改善できるか」を評価
- 複数の要素を組み合わせたスコア計算
- 0-1の範囲に正規化されたスコア

---

### 3.5 投稿パフォーマンスタグ判定アルゴリズム

**発明の構成要素**：

1. **新ロジック（再現可能性・改善可能性ベース）**：
   - 再現可能性スコア >= 0.5 の場合 → tag = "gold"（成功パターン）
   - 改善可能性スコア >= 0.5 の場合 → tag = "red"（改善が必要）
   - 上記以外の場合 → 従来ロジックにフォールバック

2. **従来ロジック（感情スコア + KPI判定ベース）**：
   - 感情スコア >= 0.65 かつ highKpi = true → tag = "gold"（成功パターン）
   - 感情スコア >= 0.65 かつ highKpi = false → tag = "gray"（主観満足は高いが指標が伸びない）
   - 感情スコア <= 0.4 かつ lowKpi = true → tag = "red"（改善が必要）
   - 上記以外 → tag = "neutral"（参考レベル）

**技術的特徴**：
- 主観的満足度（感情スコア）と客観的指標（KPI判定）を組み合わせた4段階分類
- 再現可能性・改善可能性を考慮した新しい評価ロジック
- 数値が低くても再現可能なパターンがあれば「gold」、数値が高くても改善可能な要因が特定できれば「red」とする柔軟な評価

**実装例**：
```typescript
// 新しい評価ロジック：再現可能性と改善可能性
const reproducibilityScore = calculateReproducibilityScore(record, aggregate, allRecords);
const improvabilityScore = calculateImprovabilityScore(record, aggregate, baseline);
const sentiment = deriveSentimentScore(aggregate, record.sentiment);

let tag: PostPerformanceTag = "neutral";

// 「良かった（gold）」: 再現可能なパターンがある（数値が低くてもOK）
if (reproducibilityScore >= 0.5) {
  tag = "gold";
}
// 「ダメだった（red）」: 改善可能な要因が特定できる（数値が高くてもOK）
else if (improvabilityScore >= 0.5) {
  tag = "red";
}
// 従来のロジックも参考として使用
else {
  if (sentiment.score >= 0.65 && highKpi) {
    tag = "gold";
  } else if (sentiment.score >= 0.65 && !highKpi) {
    tag = "gray";
  } else if (sentiment.score <= 0.4 && lowKpi) {
    tag = "red";
  }
}
```

### 3.6 発明のバリエーション

**バリエーション1：重み付けKPIスコア**
- 各指標（リーチ、エンゲージメント率、フォロワー増加）に重み付けを適用
- アカウントの目標に応じて、重要視する指標に高い重みを設定

**バリエーション2：時系列考慮型ベースライン**
- 単純な平均ではなく、時系列を考慮したベースライン計算
- 直近の投稿により高い重みを付与

**バリエーション3：カテゴリ別ベースライン**
- 投稿タイプ（フィード/リール/ストーリー）ごとにベースラインを算出
- カテゴリ別の相対評価を実現

本発明により、以下のようなメリットが得られる。

### 4.1 複数の指標を統合評価できる

**メリットの詳細**：
- 従来技術では、リーチ数、エンゲージメント率、フォロワー増加など、各指標を個別に表示していたが、本発明により、これらの指標を統合した単一のKPIスコアを算出できる
- ユーザーは複数の指標を見比べる必要がなく、KPIスコアを見るだけで投稿のパフォーマンスを把握できる
- どの指標が重要か、どの指標を優先すべきかを判断する必要がなくなる

**従来技術との比較**：
- **従来技術**：リーチ数は高いがエンゲージメント率が低い投稿と、リーチ数は低いがエンゲージメント率が高い投稿のどちらが「良い」投稿か判断が困難
- **本発明**：KPIスコアという単一の指標で、投稿のパフォーマンスを総合的に評価できる

---

### 4.2 ベースラインとの相対評価により、アカウント規模に依存しない評価が可能

**メリットの詳細**：
- 従来技術では、各メトリクスの絶対値のみを表示していたが、本発明により、そのアカウントの過去平均値（ベースライン）との相対評価を実現
- アカウント規模やフォロワー数に関わらず、適切な評価が可能
- 過去の投稿と比較して「良い/悪い」を自動判定できる

**従来技術との比較**：
- **従来技術**：100いいねを獲得した投稿が「良い」か「悪い」かは、そのアカウントの過去平均が50いいねなのか500いいねなのかによって異なるが、自動判定できない
- **本発明**：ベースラインとの相対評価により、100いいねが過去平均の2倍なら「良い」、0.2倍なら「悪い」と自動判定できる

---

### 4.3 投稿を自動分類できる

**メリットの詳細**：

- ユーザーは、表示されたデータを見て自分で判断する必要がなく、自動分類された結果を活用できる

本発明は、ソーシャルメディア投稿の複数の指標を統合し、ベースラインとの相対評価によりKPIスコアを算出し、主観的満足度と客観的指標を組み合わせて投稿を4段階（gold/gray/red/neutral）に自動分類するシステムである。従来技術では、各指標を個別に表示するのみで、統合評価や自動分類の機能がなかったが、本発明により、複数の指標を統合評価し、投稿を自動分類できるようになり、作業効率が大幅に向上する。また、ベースラインとの相対評価により、アカウント規模に依存しない評価が可能となり、再現可能性や改善可能性を考慮した柔軟な評価を実現する。
